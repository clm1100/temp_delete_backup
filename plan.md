# 权限控制系统需求规格说明书

## 1. 引言

### 1.1 项目背景
本项目为基于 React + Vite 技术栈的体育场馆管理 SaaS 系统。系统需支持多级权限管理，不同角色用户登录后看到的内容、可操作的功能及可访问的数据范围各不相同。

### 1.2 目标
- 实现基于角色的访问控制（RBAC），确保用户只能访问授权的页面和功能。
- 实现数据隔离，不同角色的用户只能看到与自己相关的数据（如体育馆信息、员工信息等）。
- 提升用户体验，通过前端界面隐藏无权限的菜单、按钮，减少误操作。
- 保证系统安全性，所有敏感操作均由后端进行最终权限校验。

---

## 2. 用户角色定义

| 角色名称       | 描述                                                                 | 数据范围                       |
|----------------|----------------------------------------------------------------------|--------------------------------|
| 超级管理员     | 系统最高权限，可管理所有体育馆、所有员工、系统配置。                 | 所有体育馆数据                 |
| 场馆管理员     | 负责某个具体场馆的管理工作，可管理该场馆的员工、场馆配置等。         | 仅自己所属的体育馆数据         |
| 普通员工       | 场馆内普通工作人员，只能查看部分信息，可能拥有部分操作权限（如报修、预订）。 | 仅自己所属的体育馆数据，且操作受限 |

*注：实际系统中可进一步细分权限，通过权限码（permission code）实现更细粒度的控制。*

---

## 3. 功能需求

### 3.1 登录与认证
- 用户通过用户名/密码登录系统。
- 登录成功后，后端返回用户基本信息，包括：
  - 用户ID
  - 角色标识（如 `super_admin`, `venue_admin`, `staff`）
  - 所属场馆ID（非超级管理员必填）
  - 权限码列表（可选，用于细粒度功能控制）
- 前端存储用户信息（如 localStorage），用于后续请求鉴权和界面展示。
- 提供登出功能，清除本地用户信息并跳转至登录页。

### 3.2 路由权限控制
- 系统根据用户角色动态决定可访问的路由。
- 未登录用户访问任何受保护页面时，自动重定向至登录页。
- 已登录但无权限访问某页面时，跳转至 403 无权限页面或首页。
- 路由权限配置需支持：
  - 基于角色的白名单（如 `/venues` 仅超级管理员可访问）
  - 基于权限码的白名单（可选）

### 3.3 菜单与导航权限
- 侧边栏菜单根据用户角色动态生成，无权限的菜单项不显示。
- 菜单配置包含：
  - 菜单标识、显示名称、图标、路由路径
  - 允许访问的角色列表或权限码

### 3.4 页面内权限控制（细粒度）
- 页面中的按钮、表单、数据块等组件需根据用户角色或权限码决定是否显示或启用。
- 提供可复用的权限控制组件（如 `<PermissionGuard>`）和自定义 Hook（如 `usePermission`），支持以下判断方式：
  - 角色匹配：允许一个或多个角色
  - 权限码匹配：允许拥有指定权限码的用户
  - 复合条件：同时满足角色和权限码

### 3.5 数据权限隔离
- 对于列表页、详情页等需要展示数据的场景，前端应确保请求中携带当前用户的所属场馆ID（若角色非超级管理员）。
- 后端接口需根据用户身份自动过滤数据：
  - 超级管理员可查询所有场馆数据。
  - 场馆管理员只能查询自己所属场馆的数据。
  - 普通员工只能查询自己所属场馆且被授权的数据（如只读数据）。
- 前端在展示数据时，不再额外进行过滤（后端返回即合法数据），但可根据角色对某些操作按钮做进一步隐藏。

### 3.6 权限管理后台（可选）
- 超级管理员可在后台管理：
  - 用户账号的创建、删除、禁用
  - 用户角色分配
  - 角色权限配置（如为场馆管理员分配具体的权限码）
- 场馆管理员可在后台管理：
  - 本场馆员工的账号创建、删除、禁用
  - 为本场馆员工分配权限（如普通员工只能查看报表，不能编辑）

## 4. 非功能需求

### 4.1 安全性
- 前端不得存储敏感密码等关键信息。
- 所有API请求需携带认证令牌（如JWT），令牌有效期管理。
- 前端权限控制仅用于提升用户体验，不替代后端权限校验。后端必须对所有请求进行严格的权限验证，防止用户通过伪造请求获取未授权数据。
- 前端路由守卫应防止未登录用户访问受保护页面，但后端仍需对每个API进行鉴权。

### 4.2 性能
- 权限判断逻辑需轻量高效，避免在每次渲染时进行复杂计算。
- 菜单、路由等配置应提前静态定义，避免运行时动态加载大量数据。

### 4.3 可扩展性
- 权限模型应支持未来新增角色或细粒度权限码，无需大规模重构前端代码。
- 权限配置应集中管理（如独立的配置文件），便于维护和修改。

### 4.4 用户体验
- 无权限的菜单项应隐藏，避免用户点击后出现403错误。
- 页面内的无权限按钮可置灰或隐藏，保持界面整洁。
- 登录态过期时应自动跳转至登录页并给出友好提示。

---

## 5. 业务规则

1. **角色继承**：系统暂不考虑角色继承关系，各角色权限独立。
2. **数据隔离**：场馆管理员和普通员工的数据范围均为其所属场馆，但场馆管理员拥有更多操作权限（如编辑、管理员工）。
3. **权限优先级**：当角色判断与权限码判断同时存在时，以权限码为准（细粒度覆盖粗粒度）。
4. **多场馆支持**：若未来某用户可能同时管理多个场馆，则需扩展数据权限模型，当前版本假设用户仅属于单一场馆。

---

## 6. 验收标准

- [ ] 不同角色登录后，菜单项正确显示或隐藏。
- [ ] 未授权访问路由时，正确跳转403页面。
- [ ] 同一页面内，不同角色看到的按钮/数据块符合预期。
- [ ] 场馆管理员无法看到其他场馆的数据（通过API验证）。
- [ ] 普通员工无法执行管理员专属操作（如添加员工）。
- [ ] 超级管理员可正常管理所有体育馆及用户。
- [ ] 登录态过期或登出后，无法访问任何受保护页面。
- [ ] 所有API请求均携带正确的认证信息，后端鉴权生效。
